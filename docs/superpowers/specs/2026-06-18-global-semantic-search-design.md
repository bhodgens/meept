# Global Semantic Search (TUI + Flutter + Backend) — Design Spec

**Date:** 2026-06-18
**Status:** Draft (pending user review)
**Goal:** Add a full search feature that uses semantic vector retrieval to find context across all Meept data — session messages, memories, tasks, and plans. Triggered by `f` from the sessions panel/tab in both TUI and Flutter. No per-query LLM call: embeddings are generated by a background worker and reused at query time. Memory has working semantic search already; we extend the same pattern to session messages and wire it through both UIs.

---

## Context

Meept already has keyword search across sessions/tasks/memories/plans in `internal/services/search_service.go`, exposed via HTTP (`POST /api/v1/search`). Flutter has a full `SearchPanel` wired to it. But session message **content** is not indexed at all, there is no semantic/vector search exposed beyond the memory subsystem, no RPC endpoint for TUI access, and result navigation in the Flutter UI is an unwired TODO.

## Requirements

1. **Triggers**:
   - TUI: `f` key from the sessions panel opens a search view.
   - Flutter: `f` key from the sessions tab opens search.
   - Distinct from `ctrl+f`/`cmd+f` (in-session find from Spec A).
2. **Coverage**: session messages (newly indexed), memories (already vector-indexed), tasks (keyword fallback), plans (keyword fallback).
3. **Default mode**: semantic retrieval using cosine similarity over embeddings. No per-query LLM call.
4. **Embedding generation**: background worker, no write-path latency.
5. **TUI access**: new `search.semantic` RPC method so the TUI can call it without HTTP.
6. **Flutter**: existing `SearchPanel` gets a semantic toggle (default on); result navigation wired.
7. **Graceful degradation**: if no embedding provider is configured, fall back to keyword-only search with a clear UI indicator.

## Section 1: Backend — session message FTS + vector index

### 1.1 Schema additions

In `internal/session/store_sqlite.go`, add migration creating:

```sql
-- FTS5 mirror of session_messages for keyword fallback
CREATE VIRTUAL TABLE IF NOT EXISTS session_messages_fts USING fts5(
    message_id UNINDEXED,
    session_id UNINDEXED,
    role UNINDEXED,
    content,
    tokenize = 'porter unicode61'
);

-- Triggers to keep FTS in sync with session_messages
CREATE TRIGGER IF NOT EXISTS session_messages_ai_fts AFTER INSERT ON session_messages BEGIN
    INSERT INTO session_messages_fts(message_id, session_id, role, content)
    VALUES (new.id, new.session_id, new.role, new.content);
END;
CREATE TRIGGER IF NOT EXISTS session_messages_ad_fts AFTER DELETE ON session_messages BEGIN
    DELETE FROM session_messages_fts WHERE message_id = old.id;
END;
CREATE TRIGGER IF NOT EXISTS session_messages_au_fts AFTER UPDATE ON session_messages BEGIN
    DELETE FROM session_messages_fts WHERE message_id = old.id;
    INSERT INTO session_messages_fts(message_id, session_id, role, content)
    VALUES (new.id, new.session_id, new.role, new.content);
END;
```

### 1.2 Vector table

New `vec0` virtual table for message embeddings:

```sql
CREATE VIRTUAL TABLE IF NOT EXISTS session_message_vectors USING vec0(
    message_id INTEGER PRIMARY KEY,
    embedding float[768]  -- dimension matches configured embedding model
);
```

The dimension is parameterized — populated at migration time from the configured embedding provider's dimension (default 768 for `nomic-embed-text` via Ollama, 1536 for OpenAI `text-embedding-3-small`).

### 1.3 Store interface extension

Add to `internal/session/store.go` `Store` interface:

```go
// SearchMessages performs FTS5 keyword search over session_messages.
SearchMessages(ctx context.Context, query string, limit int) ([]MessageSearchResult, error)

// SearchMessagesSemantic performs vector similarity search.
SearchMessagesSemantic(ctx context.Context, embedding []float32, limit int) ([]MessageSearchResult, error)

// StoreEmbedding persists an embedding for a message.
StoreEmbedding(ctx context.Context, messageID int64, embedding []float32) error

// UnembeddedMessages returns message IDs that have no embedding.
UnembeddedMessages(ctx context.Context, limit int) ([]int64, error)
```

```go
type MessageSearchResult struct {
    MessageID  int64   `json:"message_id"`
    SessionID  string  `json:"session_id"`
    Role       string  `json:"role"`
    Content    string  `json:"content"`
    Snippet    string  `json:"snippet"`     // truncated content with match context
    Relevance  float64 `json:"relevance"`   // 0..1
    Timestamp  string  `json:"timestamp"`
}
```

### 1.4 Keyword search implementation

```sql
SELECT sm.id, sm.session_id, sm.role, sm.content, sm.timestamp,
       bm25(session_messages_fts) AS rank
FROM session_messages_fts fts
JOIN session_messages sm ON sm.id = fts.message_id
WHERE session_messages_fts MATCH ?
ORDER BY rank
LIMIT ?;
```

Snippet generated via `snippet(session_messages_fts, 3, '<mark>', '</mark>', '...', 20)`.

### 1.5 Semantic search implementation

```sql
SELECT sm.id, sm.session_id, sm.role, sm.content, sm.timestamp, v.distance
FROM session_message_vectors v
JOIN session_messages sm ON sm.id = v.message_id
WHERE v.embedding MATCH ?
  AND k = ?
ORDER BY v.distance;
```

`distance` converted to `relevance = 1.0 - distance/2.0` (cosine distance range 0..2).

## Section 2: Backend — embedding background worker

### 2.1 Worker design

New file `internal/session/embedding_worker.go`:

```go
type EmbeddingWorker struct {
    store       Store
    embedder    memory.EmbeddingProvider
    logger      *slog.Logger
    batch       int           // messages per tick, default 20
    interval    time.Duration // default 60s
    stopChan    chan struct{}
    stopped     chan struct{}
}

func (w *EmbeddingWorker) Start() { /* ticker loop calling w.tick() */ }
func (w *EmbeddingWorker) Stop()  { /* close stopChan, wait stopped */ }

func (w *EmbeddingWorker) tick() {
    ids, err := w.store.UnembeddedMessages(ctx, w.batch)
    if err != nil { ...log and return... }
    if len(ids) == 0 { return }

    // Fetch message content for the batch
    messages := make([]string, len(ids))
    for i, id := range ids { /* fetch content */ }

    embeddings, err := w.embedder.GenerateEmbeddings(ctx, messages)
    if err != nil { ...log, return, will retry next tick... }

    for i, id := range ids {
        if err := w.store.StoreEmbedding(ctx, id, embeddings[i]); err != nil { ...log... }
    }
}
```

### 2.2 Wire into daemon

In `internal/daemon/components.go` next to memory manager initialization:

- Construct the worker if the memory manager has a non-nil embedding provider.
- Register with the scheduler for periodic runs OR start as a background goroutine with a ticker.
- Lifecycle: start after daemon init, stop on daemon shutdown (graceful with `Stop()`).

### 2.3 Embedding model resolution

Reuse the same embedding provider configured for memory in `internal/memory/manager.go`. No separate config — the assumption is that if semantic memory search is configured, session message semantic search uses the same model.

### 2.4 Graceful degradation

- No embedding provider configured → worker does not start; `SearchMessagesSemantic` returns `ErrSemanticUnavailable`; `SearchService.SearchSemantic` falls back to keyword search and sets a `Mode: "keyword"` flag in the response.
- Worker falls behind → queries still work; the FTS5 keyword search operates on all messages immediately, semantic coverage grows as the worker catches up.

## Section 3: Backend — `SearchService.SearchSemantic`

### 3.1 New method

Add to `internal/services/search_service.go`:

```go
type SemanticSearchRequest struct {
    Query string `json:"query"`
    Scope string `json:"scope,omitempty"`  // "all", "sessions", "messages", "memories", "tasks", "plans"
    Limit int    `json:"limit,omitempty"`
}

type SemanticSearchResponse struct {
    Results []SearchResult `json:"results"`
    Mode    string         `json:"mode"`  // "semantic" or "keyword" (fallback)
    Err     string         `json:"err,omitempty"`
}

func (s *SearchService) SearchSemantic(ctx context.Context, req SemanticSearchRequest) (SemanticSearchResponse, error)
```

### 3.2 Algorithm

1. Embed the query using the memory manager's embedding provider.
2. If embedding fails or no provider: fall back to keyword search, set `Mode = "keyword"`.
3. Fan out per-scope:
   - `sessions` + `messages`: call `sessionStore.SearchMessagesSemantic` (or `SearchMessages` if keyword mode).
   - `memories`: call `memoryMgr.SearchSemantic` (or `memoryMgr.Search` if keyword mode).
   - `tasks`: keyword-only (`taskRegistry.List()` + `keywordRelevance`).
   - `plans`: keyword-only (`planStore.ListPlans()` + `keywordRelevance`).
4. Merge results, sort by relevance (normalize scores to 0..1 per source), cap at `Limit`.
5. Return.

### 3.3 Scope handling

- `scope = ""` or `"all"`: search all scopes.
- `scope = "sessions"`: alias for `"messages"` (since session messages are the content).
- `scope = "messages"`: only session messages.
- `scope = "memories"`, `"tasks"`, `"plans"`: that single scope.

## Section 4: Backend — transport exposure

### 4.1 HTTP endpoint

Add to `internal/comm/http/api_handlers.go`:

```go
// POST /api/v1/search/semantic
func (h *Handlers) handleSearchSemantic(c *gin.Context) {
    var req services.SemanticSearchRequest
    if err := c.BindJSON(&req); err != nil { ... }
    resp, err := h.search.SearchSemantic(c.Request.Context(), req)
    if err != nil { ... }
    c.JSON(http.StatusOK, resp)
}
```

Register in `internal/comm/http/server.go` next to the existing `/api/v1/search` route.

### 4.2 RPC method

Register `search.semantic` in `internal/rpc/proxy.go` using the existing `makeProxy` pattern against the message bus, OR register as a direct handler since the service is in-process.

New file `internal/rpc/search.go`:

```go
func RegisterSearchHandlers(server *rpc.Server, searchSvc *services.SearchService) {
    server.RegisterHandler("search.semantic", func(ctx context.Context, params json.RawMessage) (any, error) {
        var req services.SemanticSearchRequest
        if err := json.Unmarshal(params, &req); err != nil {
            return nil, err
        }
        return searchSvc.SearchSemantic(ctx, req)
    })
}
```

Wire into `internal/daemon/components.go` alongside existing service registration.

### 5.3 Result navigation

On `enter`, dispatch based on result `Type`:
- `messages`: emit `NavigateToSessionMsg{SessionID, MessageID}` → app switches to chat view, loads session, scrolls to message.
- `memories`: switch to memory view filtered to that ID.
- `tasks`: switch to tasks view filtered to that ID.
- `plans`: switch to plans view filtered to that ID.

For the MVP, message navigation is the highest-value path; other types can show a detail modal if view-switching is complex.

### 5.4 RPC client

Add to `internal/tui/rpc.go`:

```go
func (c *RPCClient) SearchSemantic(query, scope string, limit int) (*types.SemanticSearchResponse, error) {
    result, err := c.Call("search.semantic", map[string]any{
        "query": query, "scope": scope, "limit": limit,
    })
    if err != nil { return nil, err }
    var resp types.SemanticSearchResponse
    if err := json.Unmarshal(result, &resp); err != nil { return nil, err }
    return &resp, nil
}
```

Debounced queries (250ms after last keystroke) using a `tea.Cmd` timer — pattern from `internal/tui/handlers/task_events.go`.

## Section 8: Rollout

### Migration safety

- The new `session_messages_fts` virtual table and `session_message_vectors` are additive — no existing columns touched.
- Migration runs on daemon start using the same pattern as existing migrations in `internal/session/store_sqlite.go`.
- Initial FTS population: a one-time backfill job (part of migration) inserts existing `session_messages` rows into the FTS table. For large datasets this is bounded by batched inserts to avoid lock contention.
- Initial vector population: handled by the background worker, which will catch up over time.

### Configuration

No new config keys required for the MVP — the worker reuses the memory embedding provider. If semantic memory search is already configured, semantic session search works automatically.

A future enhancement could add a `search.semantic.enabled` toggle to disable the worker independently.

### Performance

- **FTS5 keyword search**: ~5ms for 10k messages.
- **Vector search**: ~20ms for 10k vectors (HNSW via sqlite-vec).
- **Query embedding**: one embedding API call per search (~50ms local, ~200ms remote).
- **Background worker**: bounded by `batch` size; does not block message writes.

## Section 9: Out of scope

- **LLM query refinement/ranking/summarization** — future Spec C. The `SearchService.SearchSemantic` method is the natural extension point.
- **Embedding plans and tasks** — these have low volume and already work with keyword fallback. Embedding them is marginal value for now.
- **Re-embedding on content edit** — message edits are rare in Meept; re-embedding can be added if/when edit becomes a feature.
- **Search history persistence** — YAGNI.
- **Real-time search index updates** — background worker is sufficient; real-time is overkill.
- **Pagination of search results** — MVP returns top-N (default 20); infinite scroll can be added later.
